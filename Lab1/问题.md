# 1. 软件测试概念。基本定义。
**软件测试是为了发现错误而执行程序的过程，或者说，它是为了评估软件质量特性（如正确性、完整性、安全性）而对软件产品进行验证和确认的活动。**

## 核心目的：发现错误
**测试只能证明软件存在缺陷，而不能证明软件没有缺陷。**\
基本定义强调，测试的首要目的不是为了证明程序“能跑通”，而是尽可能多地发现尚未暴露的缺陷。

## 两个基本维度：验证与确认
软件测试在定义上包含两个既相关又不同的部分：

### 验证：“我们是否正确地构建了产品？”
* 检查软件是否按照规格说明书（需求文档、设计文档）正确地实现。
* 关注点：逻辑、功能、接口是否符合设计。

### 确认：“我们是否构建了正确的产品？”
* 检查软件是否满足了用户的实际需求和业务目标。
* 关注点：最终交付的功能是否对用户有价值，能否在真实环境下使用。

## 基本定义的外延
在现代软件工程语境下，软件测试的基本定义还包含以下几个关键要素：
### 不仅是事后检查：
测试不再仅仅是编码完成后的“运行并找错”。根据广义测试定义，它包含静态测试（如需求评审、代码走查）和动态测试（实际运行软件）。

### 评估质量：
测试是量化软件质量（如可靠性、易用性、性能、安全性）的主要手段。

### 降低风险：
通过测试暴露问题，为项目管理者提供决策依据（“是否具备上线条件”），从而将上线后的商业风险和用户损失降至最低。

## 总结
软件测试是以“发现缺陷”为手段，以“评估质量”和“降低风险”为目的，贯穿于软件生命周期（需求、设计、编码、运维）的系统性检查活动。


# 2. 测试目标。测试分类。
## 测试目标：
### 1. 发现缺陷
* 最经典的目标：通过执行程序，找出其中存在的错误、缺陷或失效。
* 早期发现缺陷可以大幅降低修复成本。
* 缺陷发现率是衡量测试有效性的重要指标。

### 2. 评估软件质量
测试是量化软件质量的主要手段，通过测试结果来评估：
* 功能性：是否正确实现了需求
* 可靠性：在特定条件下无故障运行的能力
* 易用性、性能、安全性、可维护性等

### 3. 验证与确认（V&V）
* 验证：确保软件“正确地实现了规格”（Build the product right）。
* 确认：确保软件“实现了用户真正需要的功能”（Build the right product）。

### 4. 降低风险
测试揭示潜在的问题，帮助项目团队判断：
* 软件是否具备上线条件
* 哪些风险尚未解决
* 为发布决策提供客观依据，从而降低业务风险、技术风险和安全风险。

### 5. 增强信心与提供信息
* 向利益相关者（管理者、客户、用户）提供关于软件状态的客观信息，帮助做出合理决策（如是否继续开发、是否发布）。

### 6. 预防缺陷
* 通过静态测试（评审、静态分析）和早期测试活动，在缺陷尚未成为代码缺陷之前就将其消除，起到预防作用。

## 测试分类

软件测试可以从多个维度进行分类，常见的分类方式如下：

### 1. 按测试阶段划分

| 分类 | 说明 |
|------|------|
| **单元测试** | 针对最小的可测试单元（函数、类、模块）进行验证，通常由开发人员执行，使用白盒技术。 |
| **集成测试** | 验证多个单元或模块之间的交互接口、数据流是否正确，关注接口和集成后的整体行为。 |
| **系统测试** | 在真实或模拟的完整系统环境中，对集成后的整体软件进行测试，验证是否符合需求规格。 |
| **验收测试** | 由用户或客户主导，确认系统是否满足业务需求、合同约定，决定是否接受系统。 |

### 2. 按是否执行代码划分

| 分类 | 说明 |
|------|------|
| **静态测试** | 不运行被测软件，通过评审、走查、审查、静态分析等手段检查文档或代码。 |
| **动态测试** | 实际运行软件，输入测试数据，观察输出结果是否符合预期。 |

### 3. 按测试技术/方法划分

| 分类 | 说明 |
|------|------|
| **黑盒测试** | 不考虑内部结构，仅基于输入和输出验证功能是否符合需求。如等价类划分、边界值分析。 |
| **白盒测试** | 基于代码内部逻辑、路径、条件设计测试用例，关注覆盖率和内部结构。如语句覆盖、分支覆盖。 |
| **灰盒测试** | 介于黑盒与白盒之间，既了解部分内部实现，又从外部进行测试。常用于集成测试或安全测试。 |

### 4. 按测试目的划分

| 分类 | 说明 |
|------|------|
| **功能测试** | 验证软件是否实现了规定的功能。 |
| **非功能测试** | 包括性能测试、压力测试、安全测试、兼容性测试、可用性测试、可靠性测试等，验证软件的质量属性。 |
| **回归测试** | 在软件修改后，重新执行已通过的测试用例，确认修改未引入新的缺陷。 |
| **冒烟测试** | 对核心功能进行快速验证，确认主要流程是否畅通，用于判断版本是否具备可测试性。 |
| **探索性测试** | 同时进行学习、设计、执行，强调测试人员的主动探索和直觉，不依赖预先写好的用例。 |

### 5. 按是否自动化划分

| 分类 | 说明 |
|------|------|
| **手工测试** | 由测试人员手动执行用例，适合探索性测试、可用性测试等。 |
| **自动化测试** | 借助工具或脚本自动执行测试，适用于回归测试、性能测试等需要反复执行的场景。 |


# 3. 单元测试。模块的概念。
**单元测试**和**模块**是软件测试与软件工程中紧密关联的两个基础概念。

## 一、模块的概念

### 1. 什么是模块
在软件工程中，**模块**是指一个**逻辑上独立、可单独命名和引用的程序组成部分**。它通常具有以下特征：

- **内聚性**：模块内部的元素（数据、函数）共同完成一个明确的功能。
- **接口清晰**：通过定义良好的输入、输出和调用方式与外部交互。
- **可独立开发/编译**：在代码组织上，模块可以是一个源文件、一个类、一个组件，或一组相关函数的集合。

### 2. 模块化的目的
- **降低复杂度**：将大系统分解为若干小模块，便于理解和维护。
- **重用性**：模块可在不同项目或系统中重复使用。
- **并行开发**：不同模块可由不同团队同时开发。
- **隔离变更**：修改一个模块的内部实现，只要接口不变，不影响其他模块。

在实际开发中，模块的体现形式因编程范式而异：
- 结构化编程：一个文件、一组函数。
- 面向对象编程：一个类或一个包（Package）。
- 组件化/微服务：一个独立的服务或库。

---

## 二、单元测试的概念

### 1. 什么是单元测试
**单元测试**是对软件中**最小可测试单元**进行验证的活动。这个“最小单元”通常是一个函数、一个方法、一个类，或者一个逻辑上不可再分的模块。

单元测试的核心目标是：
- **验证单元的行为是否符合预期**（正确性、边界条件、异常处理等）。
- **隔离依赖**：通过测试替身（如桩对象、模拟对象）隔离被测试单元，只关注该单元自身的逻辑。
- **快速反馈**：单元测试应能在几秒到几分钟内执行完毕，帮助开发人员及时发现代码改动引入的问题。

### 2. 单元测试的特点
- **由开发人员编写和维护**（通常采用测试框架，如 JUnit、pytest、Mocha 等）。
- **白盒测试为主**：基于代码内部逻辑设计测试用例，关注覆盖率和路径。
- **自动化**：通常集成在持续集成（CI）流程中，每次代码提交时自动运行。
- **频繁执行**：用于回归测试，确保修改不会破坏既有功能。


## 三、模块与单元测试的关系

| 维度 | 说明 |
|------|------|
| **测试对象** | 单元测试的“单元”通常是模块中的一个或几个紧密相关的组成部分。例如：对一个类中的每个方法进行单元测试，这个类本身就是一个模块。 |
| **隔离要求** | 单元测试要求**隔离被测试模块**，将其依赖的其他模块（如数据库、网络服务、其他类）用模拟对象替换。这保证了测试的稳定性和速度。 |
| **模块化与可测性** | 良好的模块化设计天然有利于单元测试：高内聚、低耦合的模块更容易隔离，接口清晰使得测试桩易于编写。反之，如果模块之间耦合过紧，单元测试会变得困难。 |

## 四、总结

- **模块**是从软件架构角度定义的独立、可复用的逻辑单元，强调内聚性和接口清晰。
- **单元测试**是从测试活动角度定义的对最小可测试单元的自动化验证，强调隔离和快速反馈。
- **关系**：单元测试以模块的组成部分为对象，而良好的模块化设计是单元测试能够顺利实施的前提。单元测试反过来也验证了模块的接口设计和内部逻辑的正确性。


# 4. V 型模型。静态测试和动态测试。
## 一、V型模型（V-Model）
### 1. 定义与结构
V型模型是一种**软件开发生命周期模型**，它是瀑布模型的变种，强调了**开发活动与测试活动的对应关系**。

模型呈现为字母“V”形状：
- **左侧**：代表**开发阶段**，从需求分析、系统设计、架构设计、详细设计到编码，逐步细化。
- **右侧**：代表**测试阶段**，从单元测试、集成测试、系统测试到验收测试，逐步集成和验证。
- **水平对应**：每个开发阶段都有一个对应的测试阶段，两者在同一抽象层次上相互验证。

### 2. 各阶段对应关系
| 左侧（开发） | 右侧（测试） | 说明 |
|--------------|--------------|------|
| 需求分析 | 验收测试 | 验收测试基于需求规格，确认系统是否满足用户需求。 |
| 系统设计 | 系统测试 | 系统测试基于系统设计文档，验证整体功能和非功能特性。 |
| 架构设计 | 集成测试 | 集成测试验证模块间的交互是否符合架构设计。 |
| 详细设计 | 单元测试 | 单元测试验证每个最小单元是否符合详细设计。 |
| 编码 | （单元测试具体执行） | 编码完成后，执行单元测试验证代码。 |

### 3. 核心思想
- **测试介入提前**：测试活动不是等到编码完成才开始，而是在需求、设计阶段就规划对应的测试用例。
- **质量内建**：每个开发阶段都有相应的验证手段，缺陷能够更早被发现。
- **可追溯性**：测试用例可以追溯到对应的需求或设计，确保覆盖完整。

## 二、静态测试与动态测试

### 1. 静态测试
**定义**：在不执行被测软件的情况下，通过人工评审、静态分析等方式对软件工作产品（需求文档、设计文档、代码、测试用例等）进行检查的活动。

**常见形式**：
- **正式评审**：如需求评审、设计审查、代码审查（走查、审查会议）。
- **静态分析**：借助工具（如 SonarQube、Checkstyle、PMD）检查代码是否符合编码规范、是否存在潜在缺陷（如空指针、资源泄漏）。

**特点**：
- 不需要运行程序，可以在开发早期进行。
- 能发现需求错误、设计缺陷、代码规范问题、逻辑遗漏等。
- 成本相对较低，缺陷发现早，修复成本也低。

### 2. 动态测试
**定义**：通过实际运行被测软件，输入测试数据，观察输出结果与预期是否一致的活动。

**常见形式**：
- 单元测试、集成测试、系统测试、验收测试等（即上一节提到的右侧测试活动）。
- 功能测试、性能测试、安全测试等。

**特点**：
- 需要执行软件，依赖测试环境。
- 能够验证软件在真实运行时的行为、性能、交互效果。
- 发现的是执行时才会暴露的缺陷（如崩溃、逻辑错误、性能瓶颈）。

### 3. 静态测试与动态测试的关系
- **互补性**：静态测试发现早期缺陷，动态测试验证实际运行行为。两者结合可以更全面地保障质量。
- **顺序**：通常在动态测试之前先进行静态测试，可以提前消除大量低级错误，提高动态测试的效率。
- **关注点不同**：静态测试侧重于规范性、结构、逻辑完整性；动态测试侧重于功能、性能、交互。

---

## 三、V型模型中的静态测试与动态测试

在V型模型中，静态测试与动态测试分别对应不同的阶段：

| 阶段 | 活动类型 | 说明 |
|------|----------|------|
| 需求分析 | **静态测试** | 评审需求规格说明书，确保需求清晰、一致、可测试。 |
| 系统设计 | **静态测试** | 评审系统设计文档，验证架构的合理性和完整性。 |
| 架构设计 | **静态测试** | 评审架构设计，检查模块划分、接口定义。 |
| 详细设计 | **静态测试** | 评审详细设计文档，确保单元接口和逻辑正确。 |
| 编码 | **静态测试**（代码审查、静态分析） + **动态测试**（单元测试） | 编码阶段既包含静态检查（如代码审查），也包含动态执行单元测试。 |
| 集成测试 | **动态测试** | 运行集成测试用例，验证模块间交互。 |
| 系统测试 | **动态测试** | 在完整系统上执行测试，验证功能和性能。 |
| 验收测试 | **动态测试** | 在用户环境或模拟环境下运行验收测试。 |

### 关键点
- **左侧（开发阶段）**：以静态测试为主，确保设计质量。
- **右侧（测试阶段）**：以动态测试为主，验证实现质量。
- **横跨两侧**：编码阶段同时涉及静态测试（代码审查、静态分析）和动态测试（单元测试），体现了从静态到动态的过渡。

---

## 四、总结

- **V型模型** 将开发与测试活动一一对应，强调了测试应贯穿整个生命周期，并明确了不同抽象层次上的验证关系。
- **静态测试** 是通过不运行软件的方式发现缺陷，主要在开发前期进行，效率高、成本低。
- **动态测试** 是通过运行软件验证其实际行为，主要在开发后期进行，是确认软件正确性的关键手段。
- 在实际项目中，两者结合使用，才能既保证设计的正确性，又保证实现的可靠性。V型模型恰恰体现了这种结合的框架。

# 5. 验证与确认。黑盒测试和白盒测试。
## 一、验证与确认（Verification & Validation，V&V）

验证与确认是贯穿整个软件生命周期的质量保证活动，虽然常被并称，但目的和侧重点不同。

### 1. 定义

| 概念 | 定义 |
|------|------|
| **验证** | 检查软件产品是否正确地实现了指定的需求、设计、标准。强调的是**是否符合规格**。 
| **确认** | 检查软件产品是否满足了用户的真实需求和业务目标。强调的是**是否满足用户预期**。 

### 2. 区分示例
假设用户需要一款“能在手机上登录并转账的App”：
- **验证**：检查登录功能是否按照设计文档实现了密码校验、图形验证码；转账时金额格式是否正确、接口返回格式是否符合协议。这些检查针对的是“开发出来的App与规格说明是否一致”。
- **确认**：让真实用户试用，确认登录流程是否便捷、转账操作是否符合使用习惯、最终是否满足了“手机转账”这一业务目标。如果用户觉得步骤太多、流程反人类，即便所有功能都“验证通过”，也属于“确认”失败。

### 3. 在V型模型中的体现
V型模型的左右两侧正好对应了验证与确认：
- **左侧（开发阶段）** 的评审、审查、静态分析等活动属于**验证**——检查设计、代码是否正确实现了上一阶段的要求。
- **右侧（测试阶段）** 的动态测试，尤其是**系统测试和验收测试**，更侧重于**确认**——验证最终产品是否满足用户需求。
- 严格来说，验证活动贯穿所有静态测试和部分动态测试（如单元测试验证代码与设计是否一致），而确认活动主要在系统测试及之后阶段。

### 4. 静态测试与动态测试中的V&V
- 静态测试（需求评审、设计审查、代码走查）主要是**验证**手段。
- 动态测试（单元测试、集成测试、系统测试、验收测试）既包含**验证**（如单元测试验证代码与详细设计一致），也包含**确认**（如验收测试确认满足用户需求）。


## 二、黑盒测试与白盒测试
这是按照 **测试设计方法（测试技术）** 进行的分类，关注的是测试用例的设计依据——是否基于内部结构。

### 1. 定义与特点

| 测试类型 | 定义 | 特点 | 设计依据 |
|----------|------|------|----------|
| **黑盒测试** | 将软件视为“黑盒”，不考虑内部结构和实现，仅基于输入与输出进行测试。 | 关注功能、接口、行为是否符合需求；适合系统测试、验收测试。 | 需求规格说明书、功能描述、用户故事。 |
| **白盒测试** | 基于软件的内部结构、逻辑路径、代码实现设计测试用例。 | 关注代码覆盖率、分支覆盖、路径覆盖；适合单元测试、集成测试。 | 源代码、详细设计文档、流程图。 |

### 2. 常用方法

**黑盒测试方法**：
- 等价类划分
- 边界值分析
- 决策表测试
- 状态迁移测试
- 场景法（用例法）

**白盒测试方法**：
- 语句覆盖
- 分支（判定）覆盖
- 条件覆盖
- 路径覆盖
- 循环覆盖

### 3. 联系与区别

| 维度 | 黑盒测试 | 白盒测试 |
|------|----------|----------|
| 测试者 | 通常由测试人员、用户执行 | 通常由开发人员执行 |
| 测试阶段 | 系统测试、验收测试为主 | 单元测试、集成测试为主 |
| 关注点 | 功能正确性、用户体验 | 代码逻辑、内部结构缺陷 |
| 依赖 | 需求文档 | 代码、设计文档 |
| 缺陷发现 | 功能缺失、接口错误、业务逻辑问题 | 逻辑错误、未覆盖路径、内存泄漏 |

在实际项目中，黑盒与白盒通常**结合使用**：单元测试阶段以白盒为主，系统测试阶段以黑盒为主，集成测试阶段则可能采用灰盒（介于两者之间）方法。

---

## 三、两组概念的关系

### 1. V&V 与 黑盒/白盒
- **验证** 既可以通过白盒方法（如单元测试、代码审查）实现，也可以通过黑盒方法（如集成测试中的接口测试）实现。
- **确认** 几乎总是采用黑盒方法（因为确认关注的是用户视角的行为，而非内部实现）。

### 2. 静态/动态测试 与 黑盒/白盒
- 静态测试（如评审）不属于黑盒或白盒的动态分类，它是一种独立的检查方式。
- 动态测试才涉及黑盒/白盒的选择。

## 四、总结

| 概念对 | 核心区别 | 典型应用 |
|--------|----------|----------|
| **验证 vs 确认** | 验证：“按规格做对了没有？”<br>确认：“做的东西对不对？” | 验证：需求评审、单元测试<br>确认：验收测试、用户试用 |
| **黑盒 vs 白盒** | 黑盒：不看内部结构，基于输入输出<br>白盒：基于内部逻辑设计用例 | 黑盒：系统测试、功能测试<br>白盒：单元测试、覆盖率分析 |


# 6. 测试用例、测试场景和测试覆盖率。
## 一、测试用例

### 定义
**测试用例**是为了某个特定测试目标而设计的一组输入数据、执行条件、操作步骤和预期结果的集合。它是测试执行的最小单元。

### 组成要素
- **标识信息**：编号、名称、所属模块
- **前置条件**：执行前必须满足的状态（如已登录、数据已准备）
- **测试数据**：具体的输入值或参数
- **操作步骤**：执行测试的详细操作序列
- **预期结果**：软件应产生的正确输出或状态变化
- **实际结果**（执行后填写）
- **测试结果**（通过/失败/阻塞）

### 作用
- 保证测试的可重复性
- 明确测试通过/失败的标准
- 便于回归测试和自动化
- 量化测试进度（已执行/通过/失败数量）


## 二、测试场景

### 定义
**测试场景**是从用户实际使用角度出发，描述一个完整的业务流程、操作路径或用户故事的测试设计。一个测试场景通常由多个相关联的测试用例串联而成，模拟真实用户在系统中的行为轨迹。

### 与测试用例的区别
| 维度 | 测试用例 | 测试场景 |
|------|----------|----------|
| 数量关系 | 一个场景包含多个用例 | 多个用例构成一个场景 |
| 示例 | “输入正确密码，验证登录成功” | “用户登录 → 搜索商品 → 加入购物车 → 下单 → 支付” |

### 作用
- 模拟真实用户操作，发现集成和业务流程中的问题
- 帮助梳理需求，确保关键业务路径被覆盖
- 用于探索性测试、验收测试、用户验收测试（UAT）
- 作为设计测试用例的蓝图

---

## 三、测试覆盖率

### 定义
**测试覆盖率**是衡量测试对被测对象覆盖程度的度量指标，用于评估测试的充分性和质量。

### 常见类型

#### 1. 需求覆盖率
- **定义**：已被测试覆盖的需求数量占总需求数量的比例。
- **意义**：确保所有业务需求都有对应的测试用例。

#### 2. 代码覆盖率（白盒测试常用）
- **语句覆盖**：被执行的代码行数比例。
- **分支/判定覆盖**：每个分支（如if/else）至少被执行一次的比例。
- **条件覆盖**：每个布尔条件取真和假均被覆盖的比例。
- **路径覆盖**：所有可能的执行路径被覆盖的比例（成本最高）。
- **函数/方法覆盖**：被调用过的函数比例。

#### 3. 接口覆盖率
- 被测试的API接口数量占总接口数量的比例。

#### 4. 场景覆盖率
- 已设计测试场景的关键业务流程占全部核心流程的比例。

### 作用
- 发现测试盲区，指导补充测试用例
- 作为质量门禁（例如要求代码覆盖率不低于80%）
- 客观评估测试工作的完整性

---

## 四、三者关系

1. **层次递进**  
   - 测试场景从用户角度定义了“测什么流程”。  
   - 测试用例从技术角度明确了“如何测、用什么数据、期望什么结果”。  
   - 测试覆盖率则回答了“测了多全”。

2. **相互支撑**  
   - 测试场景帮助设计出更有价值的测试用例集。  
   - 测试用例的执行结果可以用来计算覆盖率（例如代码覆盖率由运行用例的代码痕迹统计）。  
   - 覆盖率数据反过来指导补充测试用例，提升场景覆盖度。

## 五、总结

| 概念 | 核心回答 | 作用 |
|------|----------|------|
| **测试用例** | 最小执行单元，包含输入、步骤、预期结果 | 可重复执行、明确通过标准、支持自动化 |
| **测试场景** | 端到端的业务流程描述，由多个用例组成 | 模拟真实用户、验证业务流程完整性 |
| **测试覆盖率** | 度量测试完整性的指标 | 评估充分性、指导补充测试、设定质量门禁 |

三者共同构成测试工作的基础：**场景**定义方向，**用例**落地执行，**覆盖率**量化质量。合理使用这三个概念有助于设计高效、充分的测试方案。



# 7. 等价性分析。
**等价性分析**（在软件测试中通常称为**等价类划分**，Equivalence Partitioning）是一种黑盒测试用例设计方法。其核心思想是：将输入数据划分为若干等价类，使得同一等价类中的任意一个数据在测试中具有相同的效果（即对程序的执行路径和结果的影响相同），因此只需要从每个等价类中选取一个代表性数据作为测试用例，即可达到覆盖该类的目的。

## 一、等价类的概念
### 1. 等价类的定义
- **等价类**是指输入域（或输出域）中具有相同行为特征的子集。
- 如果程序对某一类输入的处理方式相同，那么这些输入就属于同一个等价类。

### 2. 有效等价类与无效等价类
- **有效等价类**：符合需求规格、程序应能正确处理的输入。
- **无效等价类**：不符合需求规格、程序应能正确拒绝或给出错误提示的输入。

通常，测试既要覆盖有效等价类（验证正常功能），也要覆盖无效等价类（验证容错性）。


## 二、等价类划分的原则

1. **如果输入条件规定了取值范围**（如 `1 ≤ x ≤ 100`），则：
   - 一个有效等价类：取值范围内的任意值（如 50）
   - 两个无效等价类：小于范围的值（如 0）、大于范围的值（如 101）

2. **如果输入条件规定了输入值的集合**（如 `{A, B, C}`），则：
   - 一个有效等价类：集合内的任意元素（如 A）
   - 一个无效等价类：集合外的任意值（如 D）

3. **如果输入条件规定了“必须为真”的条件**（如“首字符必须为字母”），则：
   - 一个有效等价类：满足条件（如 `a`）
   - 一个无效等价类：不满足条件（如 `1`）

4. **如果输入条件规定了输入数据是一组值（如枚举）**，则每个值可视为一个有效等价类。

5. **如果输入条件规定了输入数据必须满足某种规则（如邮箱格式）**，则：
   - 有效等价类：符合规则的任意值
   - 无效等价类：多种违反规则的情况（如缺少@、缺少域名、包含非法字符等），通常需要进一步细分。


## 三、测试用例设计步骤
1. **识别输入条件**：从需求、规格说明中提取所有输入域（包括显式输入和隐式环境条件）。
2. **为每个输入条件划分等价类**：按照上述原则确定有效等价类和无效等价类。
3. **为每个等价类编号**。
4. **设计测试用例**：
   - 覆盖尽可能多的有效等价类（但要注意避免遗漏组合关系）。
   - 为每个无效等价类单独设计测试用例（因为无效输入通常会触发异常处理，相互干扰时不易定位）。


## 四、示例说明

假设一个登录功能，用户名要求：
- 长度 4~12 个字符
- 只能包含字母和数字

**等价类划分**：

| 输入条件 | 有效等价类 | 无效等价类 |
|----------|------------|------------|
| 长度 | 4~12 字符（例如 `abc123`） | 1. 长度 <4（如 `ab`）<br>2. 长度 >12（如 `a`×13） |
| 字符类型 | 字母和数字（如 `abc123`） | 3. 包含特殊字符（如 `abc@123`）<br>4. 包含中文或其他非字母数字 |

**测试用例**（可组合有效类，无效类单独）：
- TC1：有效长度 + 有效字符（覆盖有效类）
- TC2：长度 <4（无效类1）
- TC3：长度 >12（无效类2）
- TC4：包含特殊字符（无效类3）
- TC5：包含中文（无效类4）

## 五、优点与局限性
### 优点
- **高效**：用较少的测试用例覆盖大量的输入可能性。
- **系统化**：为测试设计提供了明确的指导，避免随机测试。
- **易于理解**：方法直观，适合在需求分析阶段同步设计。

### 局限性
- **忽略边界**：等价类划分不强调边界值，而边界往往是缺陷高发区。因此通常需要与**边界值分析**结合使用。
- **依赖经验**：划分等价类需要测试人员对需求有准确理解，若划分不当则可能遗漏缺陷。
- **不考虑组合**：当多个输入条件相互依赖时，单纯等价类划分可能无法覆盖组合情况，需配合判定表或正交试验等方法。

## 总结
**等价性分析（等价类划分）** 是黑盒测试中最基础、最常用的方法之一。它将无限的输入空间划分为有限的等价类，通过选取代表性数据进行测试，在保证充分性的前提下大幅减少测试用例数量。实际应用中，等价类划分通常与边界值分析搭配使用，以发现更多的隐藏缺陷。


# 8. 决策表和转换表。
## 一、决策表
### 1. 定义
决策表是一种用于描述**逻辑条件组合与对应动作**的表格工具。它适用于业务规则复杂、输入条件之间存在多种组合的场景，能够系统化地列出所有可能的情况，避免遗漏。

### 2. 结构
决策表由四部分组成：

| 组成部分 | 说明 |
|----------|------|
| **条件桩** | 列出所有相关的条件（通常是布尔条件或状态）。 |
| **动作桩** | 列出所有可能执行的操作。 |
| **条件项** | 为每个条件赋值（真/假、Y/N、具体值等），构成一条规则。 |
| **动作项** | 在特定条件组合下应执行的动作（打勾或写明动作）。 |

通常，决策表的一列称为一条**规则**，代表一组条件组合及其对应的动作。

### 3. 示例
假设一个“申请信用卡”的规则：
- 条件1：年龄 ≥ 18 岁
- 条件2：有稳定收入
- 条件3：无不良信用记录
- 动作A：批准申请
- 动作B：拒绝申请

决策表：

| 条件/动作 | 规则1 | 规则2 | 规则3 | 规则4 | 规则5 |
|-----------|-------|-------|-------|-------|-------|
| 年龄≥18 | Y | Y | Y | N | N |
| 有稳定收入 | Y | Y | N | Y | N |
| 无不良信用 | Y | N | Y | Y | N |
| **批准** | ✔ |   |   |   |   |
| **拒绝** |   | ✔ | ✔ | ✔ | ✔ |

### 4. 测试应用
- 每条规则对应一个测试用例（或一组用例）。
- 适用于业务规则复杂、条件组合多的功能，如保险计价、审批流程、游戏规则等。
- 可结合等价类和边界值细化条件取值。

---

## 二、转换表（状态转换表）

### 1. 定义
状态转换表（State Transition Table）是一种描述**状态机**行为的表格工具，它列出了系统可能所处的状态、引发状态变化的事件、以及事件触发后的状态迁移和产生的动作。

它通常与**状态转换图**配合使用，用于测试具有明确状态和事件驱动的系统。

### 2. 结构
常见的状态转换表包含以下列：

| 当前状态 | 事件 | 下一状态 | 动作（可选） |
|----------|------|----------|--------------|
| 状态A | 事件1 | 状态B | 执行某操作 |
| 状态A | 事件2 | 状态A | 无变化，可能提示 |
| 状态B | 事件1 | 状态C | … |

### 3. 示例
一个简单的登录状态机：
- 状态：未登录、登录中、已登录
- 事件：输入用户名密码、验证成功、验证失败、登出

状态转换表：

| 当前状态 | 事件 | 下一状态 | 动作 |
|----------|------|----------|------|
| 未登录 | 输入用户名密码 | 登录中 | 显示验证中 |
| 登录中 | 验证成功 | 已登录 | 跳转主页 |
| 登录中 | 验证失败 | 未登录 | 显示错误提示 |
| 已登录 | 登出 | 未登录 | 清除会话 |

### 4. 测试应用
- 根据状态转换表设计测试用例，覆盖**状态-事件-状态**的路径。
- 关注：正常路径、异常事件（如无效事件）、状态覆盖、转换覆盖。
- 适用于具有清晰状态流转的系统，如电梯控制、订单状态（待支付→已支付→已发货→已完成）、工作流引擎等。

---

## 三、决策表与转换表的比较

| 维度 | 决策表 | 转换表 |
|------|--------|--------|
| **关注点** | 条件组合与动作的映射 | 状态、事件及状态迁移 |
| **适用场景** | 业务规则复杂、无状态记忆的逻辑 | 系统状态明确、事件驱动的行为 |
| **表现形式** | 矩阵形式，列代表规则 | 表格形式，行代表迁移 |
| **测试设计** | 每条规则生成测试用例 | 每个转换路径生成测试用例 |
| **典型应用** | 计费规则、审批逻辑、配置策略 | 登录状态、订单流程、协议栈 |

### 联系
两者都是结构化的测试设计技术，能够系统化地减少遗漏，并且在某些复杂系统中可以结合使用：例如一个订单状态转换表中，每个状态下的业务规则可能用决策表来描述。

---

## 四、总结

- **决策表**：专注于“条件 → 动作”的组合逻辑，通过枚举所有条件组合确保业务规则的完整性，适合处理“如果……那么……”的多条件分支。
- **转换表**：专注于“状态 → 事件 → 新状态”的动态行为，描述系统在不同状态下的响应和迁移，适合测试状态机、工作流、交互式系统。

在实际测试中，根据被测系统的特性选择合适的表，或两者结合，可以系统化地设计出高覆盖率的测试用例。


# 9. 回归测试。
**回归测试**（Regression Testing）是软件测试中一项至关重要的活动，它指在软件发生变更（如新增功能、修复缺陷、优化性能、配置更改等）后，重新执行之前已通过的测试用例，以验证这些变更没有引入新的缺陷，也没有破坏原有的正常功能。

## 一、为什么需要回归测试
在软件开发过程中，变更往往会带来“副作用”：
- 修复一个缺陷可能破坏其他正常功能。
- 新增代码可能影响已有模块的交互逻辑。
- 重构或优化可能引入隐藏的错误。

回归测试的核心目的就是**确保软件在演化过程中，原有功能仍然正确**，从而维护软件质量的稳定性。

## 二、何时执行回归测试
回归测试通常在以下场景触发：
- **版本迭代**：每次发布新版本前，对关键功能进行回归。
- **缺陷修复**：修复一个Bug后，验证修复有效性的同时，验证相关模块未受影响。
- **代码重构**：重构后必须确保原有行为不变。
- **环境变更**：依赖库升级、数据库变更、配置修改等。
- **持续集成**：每次代码提交自动触发回归测试（通常为自动化测试套件）。


## 三、回归测试的类型与策略

根据项目规模和资源，回归测试可以采用不同策略：

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **完全回归** | 执行所有已有的测试用例（包括功能、性能、安全等） | 关键系统、变更影响范围大、资源充足 |
| **部分回归** | 仅执行与被变更代码相关或受影响模块的测试用例 | 大多数日常迭代，基于影响分析选择 |
| **选择性回归** | 通过代码覆盖分析、变更影响分析等算法，智能选择受影响的测试用例 | 大型系统，追求效率与覆盖的平衡 |

在实际项目中，**自动化回归测试**是主流做法，因为手动重复执行大量用例成本过高且容易出错。

## 四、回归测试的自动化
自动化回归测试通常包括：
- **单元测试**：运行所有单元测试用例，快速反馈代码变更是否破坏底层逻辑。
- **接口测试**：自动化验证API的正确性。
- **UI自动化测试**：对核心业务流程进行自动化验证（如登录、下单、支付等）。
- **性能回归测试**：对比变更前后的性能指标（响应时间、吞吐量），防止性能劣化。

自动化回归测试需要与**持续集成（CI）**工具（如Jenkins、GitLab CI、GitHub Actions）结合，在代码提交时自动触发，确保尽早发现问题。

## 五、回归测试的挑战与最佳实践

### 挑战
1. **测试套件膨胀**：随着项目迭代，回归测试用例不断积累，执行时间变长。
2. **维护成本高**：UI自动化测试容易因界面变化而失效，需要持续维护。
3. **覆盖不全**：影响分析不准确可能导致遗漏受影响的模块。
4. **资源冲突**：在快速迭代中，回归测试可能成为发布瓶颈。

### 最佳实践
- **分层回归**：建立测试金字塔（单元测试 > 接口测试 > UI测试），让大部分回归测试在底层快速完成。
- **测试优先级**：将核心业务流程、高风险模块的测试用例优先纳入回归集。
- **增量回归**：结合变更影响分析，只运行与变更相关的测试，而非全量。
- **持续维护**：定期清理冗余、低效的测试用例，保持回归套件轻量化。
- **可靠的环境**：回归测试应在独立的测试环境中运行，避免因环境差异导致误报。
- **结果分析**：对失败的回归测试快速分析，判断是真正缺陷还是测试用例本身失效。

## 六、总结

回归测试不是一次性的活动，而是伴随整个软件生命周期的**质量保障屏障**。它的价值在于：
- **防止回归缺陷**，确保软件演进的稳健性。
- **提供快速反馈**，帮助开发人员及时修复问题。
- **增强发布信心**，通过可重复的验证证明原有功能未受影响。

在DevOps实践中，回归测试的自动化程度和执行效率直接影响交付速度和质量。合理规划回归策略、维护健康的测试集，是实现高质量持续交付的基础。



# 10. JUnit5 库。API 特性。junit.framework 类。
JUnit 5 是 Java 单元测试框架的最新主要版本，相比 JUnit 3/4 在架构、API 设计和扩展能力上有了根本性的变化。而 `junit.framework` 是 JUnit 3 的核心包，在 JUnit 5 中已不再作为主 API 存在，而是通过 **JUnit Vintage** 模块提供兼容支持。下面分别说明 JUnit 5 的库结构、API 特性以及与旧版 `junit.framework` 的关系。

## 一、JUnit 5 架构与模块组成

JUnit 5 由三个核心子项目组成，实现了高度模块化：

| 模块 | 作用 |
|------|------|
| **JUnit Platform** | 测试框架的底层平台，用于在 JVM 上启动测试框架。它定义了 `TestEngine` 接口，任何测试框架（JUnit Jupiter、JUnit Vintage、TestNG 等）都可以通过实现该接口集成到平台中。 |
| **JUnit Jupiter** | 新的编程模型和扩展模型的组合，提供了 JUnit 5 的核心注解、断言、动态测试等新 API。这是编写 JUnit 5 测试时主要使用的模块。 |
| **JUnit Vintage** | 兼容层，用于在 JUnit Platform 上运行基于 JUnit 3 或 JUnit 4 编写的测试。通过该模块，可以平滑迁移旧测试而不需要立即重写。 |

---

## 二、JUnit Jupiter 的主要 API 特性

### 1. 注解驱动的编程模型
JUnit 5 使用全新的注解，相比 JUnit 4 更加清晰且语义明确：

| 注解 | 说明 |
|------|------|
| `@Test` | 标记测试方法。不再需要 `public` 修饰符，可以使用包级或 `protected`。 |
| `@BeforeEach` / `@AfterEach` | 替代 JUnit 4 的 `@Before` / `@After`，在每个测试方法前后执行。 |
| `@BeforeAll` / `@AfterAll` | 替代 `@BeforeClass` / `@AfterClass`，在所有测试方法前后执行一次。方法必须是静态的（除非使用 `@TestInstance` 改变生命周期）。 |
| `@Disabled` | 替代 `@Ignore`，禁用测试类或方法。 |
| `@Nested` | 允许在测试类内部定义嵌套类，用于组织相关测试，并共享外部类的状态。 |
| `@Tag` | 用于声明标签，便于在构建工具中筛选测试（类似 JUnit 4 的 `@Category`）。 |
| `@TestFactory` | 声明动态测试的工厂方法，返回 `DynamicNode` 集合，支持运行时生成测试。 |
| `@DisplayName` | 为测试类或方法提供自定义显示名称（支持表情符号、空格等）。 |
| `@RepeatedTest` | 指定重复执行测试的次数。 |

### 2. 断言增强
`org.junit.jupiter.api.Assertions` 类提供了丰富的静态断言方法，并支持 **Lambda 表达式**来延迟消息构造：

```java
import static org.junit.jupiter.api.Assertions.*;

assertAll("group",
    () -> assertEquals(1, 1),
    () -> assertTrue(true)
);
assertThrows(IllegalArgumentException.class, () -> { ... });
assertTimeout(Duration.ofSeconds(1), () -> { ... });
assertLinesMatch(expectedLines, actualLines);
```

### 3. 参数化测试
通过 `@ParameterizedTest` 注解，可以方便地使用多种参数源：

```java
@ParameterizedTest
@ValueSource(ints = {1, 2, 3})
void testWithValueSource(int arg) { ... }

@ParameterizedTest
@CsvSource({"foo, 1", "bar, 2"})
void testWithCsvSource(String first, int second) { ... }

@ParameterizedTest
@MethodSource("stringProvider")
void testWithMethodSource(String arg) { ... }
```

### 4. 动态测试
`@TestFactory` 允许在运行时生成测试用例，适合需要动态创建测试的场景（如数据驱动测试）：

```java
@TestFactory
Stream<DynamicTest> dynamicTests() {
    return Stream.of("A", "B", "C")
        .map(input -> dynamicTest("test " + input, () -> {
            // 测试逻辑
        }));
}
```

### 5. 扩展模型（Extension Model）
JUnit 5 放弃了 JUnit 4 的 `@Rule` 和 `@ClassRule`，采用统一的 `Extension` 接口，通过组合方式实现功能扩展。常用的扩展有：

- `@ExtendWith`：声明式注册扩展。
- `BeforeAllCallback`、`BeforeEachCallback`：生命周期回调。
- `ParameterResolver`：为测试方法参数提供解析（如注入依赖）。
- `TestExecutionExceptionHandler`：处理异常。

Spring Boot 测试中常见的 `@SpringBootTest` 就是通过 `@ExtendWith(SpringExtension.class)` 实现的。

### 6. 测试实例生命周期
默认情况下，JUnit 5 为每个测试方法创建一个新的测试类实例（与 JUnit 4 相同）。但可以通过 `@TestInstance(Lifecycle.PER_CLASS)` 改为每个测试类只创建一个实例，从而允许 `@BeforeAll` / `@AfterAll` 方法不需要是静态的，并在嵌套类间共享状态。

---

## 三、junit.framework 类在 JUnit 5 中的定位

### 1. junit.framework 的历史
`junit.framework` 是 JUnit 3 的核心包，包含了以下关键类：
- `TestCase`：所有测试类必须继承的基类。
- `TestSuite`：用于组合多个测试。
- `Assert`：提供断言方法（`assertEquals`、`assertTrue` 等）。

JUnit 4 引入了注解驱动的模型，但仍然保留了 `junit.framework` 以保持向后兼容。许多遗留项目依然有基于 `TestCase` 的测试类。

### 2. 在 JUnit 5 中的兼容
JUnit 5 **不包含** `junit.framework` 包。但通过 **JUnit Vintage** 模块，可以运行基于 JUnit 3/4 的旧测试。具体机制：
- JUnit Vintage 提供了 `VintageTestEngine`，它能识别并执行 JUnit 3 的 `TestCase` 子类以及 JUnit 4 的 `@Test` 注解。
- 在构建工具（如 Maven/Gradle）中引入 `junit-vintage-engine` 依赖后，这些旧测试会像 JUnit 5 测试一样被平台发现和执行。

因此，如果项目中存在继承 `junit.framework.TestCase` 的测试类，它们依然可以在 JUnit 5 环境中运行，但新编写的测试应当使用 JUnit Jupiter 的 API。

---

## 四、迁移建议

对于从 JUnit 4（或更早）迁移到 JUnit 5 的项目：
- 可以同时使用 JUnit Vintage 和 JUnit Jupiter，实现渐进式迁移。
- 新测试使用 JUnit Jupiter 注解（`org.junit.jupiter.api`）。
- 旧测试逐步重写为 Jupiter 风格，或保持原样由 Vintage 执行。
- 如果完全放弃旧测试，可移除 `junit-vintage-engine` 依赖。

---

## 五、总结

| 方面 | JUnit 5（Jupiter） | junit.framework（JUnit 3） |
|------|---------------------|----------------------------|
| 包结构 | `org.junit.jupiter.api` | `junit.framework` |
| 测试类 | 无需继承基类，通过注解标识 | 必须继承 `TestCase` |
| 断言 | `Assertions` 类（静态导入） | `Assert` 类 |
| 初始化/清理 | `@BeforeEach` / `@AfterEach` | `setUp()` / `tearDown()` |
| 忽略测试 | `@Disabled` | 重写 `suite()` 或注释 |
| 扩展机制 | `Extension` API（声明式） | `TestDecorator` 或覆盖方法 |
| 运行平台 | JUnit Platform | 原生 JUnit 3 Runner |

JUnit 5 通过模块化设计、Lambda 支持、强大的扩展模型和丰富的注解，为现代 Java 测试提供了更灵活、更高效的编程体验。对于使用 `junit.framework` 的遗留测试，JUnit Vintage 保证了平滑过渡。
